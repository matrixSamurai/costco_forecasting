# Costco Route Delay Forecasting — Project Summary

## What This Project Does

This system predicts **how much money Costco loses when severe weather delays fresh lettuce shipments** from Tracy, CA to destination warehouses. It does this in real time as a user selects a route and departure date, outputting dollar-level revenue loss estimates and actionable shipment decisions.

The core problem: fresh lettuce has a shelf life of 5–14 days (depending on variety and temperature). A weather-induced transit delay doesn't just slow the truck — it consumes shelf life, raises spoilage rates, and directly reduces the value of the cargo. The further along the route severe weather strikes, the more shelf life has already been consumed before the delay even begins.

---

## System Architecture

```
User selects: Origin → Destination + Departure date/time + Cargo details
                        ↓
              Google Directions API  →  Up to 3 route alternatives
                        ↓
              ~20-mile pitstop sampling along each route
                        ↓
              For each pitstop: nearest NOAA weather station lookup
                        ↓
        ┌─────────────────────────────────────────────────┐
        │  PRE-MODEL (Classifier)                         │
        │  "Will there be severe weather at this point?"  │
        └──────────────┬──────────────────────────────────┘
                       │
           No severe   │   Severe weather likely
           weather     │
           delay = 0   │
                       ↓
        ┌─────────────────────────────────────────────────┐
        │  MODEL 1 (Delay Regression)                     │
        │  "How much delay (%) does this weather cause?"  │
        └──────────────┬──────────────────────────────────┘
                       ↓
        ┌─────────────────────────────────────────────────┐
        │  MODEL 2 (Loss Regression)                      │
        │  "What fraction of the cargo will spoil?"       │
        └──────────────┬──────────────────────────────────┘
                       ↓
        ┌─────────────────────────────────────────────────┐
        │  SUPPLY RISK + RECOMMENDATIONS ENGINE           │
        │  "What should we do about it?"                  │
        └─────────────────────────────────────────────────┘
                       ↓
              Revenue loss in dollars + Shipment decision
```

## Stage 0 — Route & Weather Ingestion

### What
The user picks an origin city and a Costco destination warehouse. The system queries the **Google Directions API** for up to 3 alternative routes, each represented as a polyline.

### Why
Different routes have meaningfully different weather exposure. A longer route through the Sierra Nevada may pass through more severe weather than a shorter coastal route. Giving the user 3 options lets them compare risk vs. travel time.

### How
- `route_utils.py` calls `fetch_directions()` → Google Directions API
- Each route's polyline is decoded with `decode_polyline()`
- `sample_route_points_by_distance()` places a pitstop every ~20 miles (32.2 km) along the decoded points using Haversine distance interpolation
- For each pitstop, `get_weather_features_for_pitstop(lat, lng)` finds the **nearest NOAA weather station** from `weather_stations.json` and retrieves weekly-aggregated weather features from `point_weekly_weather.json`

### Weather features extracted per pitstop (9 features)

| Feature | Description |
|---------|-------------|
| `temp_min_mean` | Min temperature (°F) |
| `temp_max_mean` | Max temperature (°F) |
| `temp_mean_mean` | Mean temperature (°F) |
| `snow_depth_mean` | Snow depth (cm) |
| `prcp_total_mean` | Precipitation (mm/day) |
| `visibility_mean` | Visibility (miles) |
| `wind_speed_mean` | Mean wind speed (mph) |
| `wind_speed_max_mean` | Max wind speed (mph) |
| `wind_gust_max_mean` | Max wind gust (mph) |

## Stage 1 — Pre-Model: Severe Weather Classifier

### What
A binary classifier that answers: **"Will this pitstop experience severe weather (fog, snow/ice, hail, or thunderstorm)?"** It runs on every pitstop and gates whether the delay regression model (Model 1) runs at all.

### Why
Model 1 is a regression model trained on **synthetic delay data** generated by a formula. It has no direct knowledge of whether real-world severe weather events actually occurred. The Pre-Model is trained on **4.34 million real NOAA observations (2021–2025)**, providing an independent, empirically grounded signal. If the Pre-Model says no severe weather, forcing Model 1 to predict a delay would be noise — so it's skipped entirely and delay is set to zero.

This design avoids false positives: mild weather days don't generate phantom delay predictions.

### Training Data
- **Source**: `data/source/weather_data_2021_2025.csv` — 4.34M raw NOAA records, 2.32M after cleaning
- **Label**: `has_severe_weather = 1` if any of: fog, snow/ice pellets, hail, or thunder was recorded
- **Class distribution**: 22.4% positive (severe), 77.6% negative — class weights applied during training

### Models Trained

| Model | Accuracy | Precision | Recall | F1 | ROC-AUC |
|-------|----------|-----------|--------|-----|---------|
| **XGBoost** (default) | 82.4% | 57.6% | 81.9% | 67.6% | **90.5%** |
| Random Forest | 81.8% | 56.5% | 81.6% | 66.8% | 90.0% |
| LightGBM | 82.3% | 57.3% | 81.9% | 67.5% | 90.4% |

Recall is optimized over precision — it is better to flag a potential delay that doesn't materialize than to miss a real one.

### Features (10)
Same 9 weather features above, plus `month` and `day_of_year` (seasonal context).

### Top feature importances
1. `visibility` — 52.4% (most discriminating: fog/snow dramatically reduce visibility)
2. `temp_mean` — 12.0%
3. `prcp_total` — 10.5%

### Route-level aggregation (in `app.py`)
After classifying every pitstop across all routes, the system computes 4 KPIs:

| KPI | Calculation |
|-----|-------------|
| **Route Status** | `True` if ≥20% of pitstops have prob ≥ 0.5, OR any pitstop has prob ≥ 0.7 |
| **Avg Probability** | Mean of all pitstop probabilities |
| **Max Probability** | Highest single pitstop probability |
| **Severe Pitstops** | Count of pitstops where probability ≥ 0.5 |

### Files
- `predict_has_delay.py` — prediction interface
- `build_classifier.py` — training script
- `classifier_models/classifier_xgboost.joblib` — trained artifact (961 KB)

## Stage 2 — Model 1: Delay Regression

### What
Predicts **what percentage delay** a shipment will experience relative to the baseline 24-hour transit time, given weather conditions and departure timing.

### Why
Not all severe weather creates equal delay. A light snowstorm at noon in June delays a truck far less than a blizzard at 2 AM in January. Model 1 is designed to capture both the **weather severity** and the **temporal context** (time of day, day of year, winter severity) simultaneously.

### Training Data
- **Source**: `data/routes/routes_with_weather_and_substation_time.json` — real route segments enriched with NOAA weather
- **Labels (synthetic)**: Delay % is generated by a formula that applies:
  - Base delay from weather (visibility, snow depth, precipitation, wind)
  - Winter severity premium (Dec/Jan = +40%, Mar = +20%, summer = 0%)
  - Night premium (midnight = +15%, noon = 0%)
  - Additive Gaussian noise for realism
- **Training size**: Up to 120,000 route segments

### Features (16)
9 weather features + 7 temporal features:

| Temporal Feature | Description |
|-----------------|-------------|
| `journey_start_hour` | Hour of day when trip begins (0–23) |
| `journey_start_day_of_week` | Day of week (0=Mon … 6=Sun) |
| `journey_start_month` | Month (1–12) |
| `week_of_year` | ISO week (1–52) |
| `day_of_year` | Day of year (1–365) |
| `winter_severity` | 0–1 scalar: Dec/Jan=1.0, Mar=0.6, Jun=0.0 |
| `segment_start_hour` | Actual clock time when the driver reaches *this* segment (accounts for cumulative drive time) |

The `segment_start_hour` feature is key: for a long route, a driver who departs at 8 AM may reach a mountain pass at 11 PM — that segment gets a night penalty even though the journey started in daylight.

### Models Trained
- **Ridge** — linear baseline (`delay_model_ridge.joblib`, 1.8 KB)
- **Random Forest** (`delay_model_random_forest.joblib`, 24 MB)
- **XGBoost** (`delay_model_xgboost.joblib`, 476 KB) — default for the pipeline

### Output
`delay_pct` (0–50%) per model, then converted to hours:
```
delay_hours = baseline_hours × (delay_pct / 100)
            = 24h × 0.30 = 7.2 hours
```

### Files
- `predict_delay.py` — prediction interface
- `build_delay_model.py` — training script

## Stage 3 — Model 2: Revenue Loss Regression

### What
Predicts **what fraction of the cargo will spoil** (`loss_rate`, 0–1) given the delay, temperature, lettuce type, and seasonal context. Multiplied by cargo value to get dollar revenue loss.

### Why
Delay hours alone don't determine spoilage. A 24-hour delay at 32°F has a very different impact than a 24-hour delay at 55°F. Shelf life varies by lettuce variety — spring mix lasts 3–5 days at 55°F but 10–14 days at 32°F. Iceberg is more robust than butterhead. Model 2 captures all these interactions in a single learned function.

The underlying decay model is a **sigmoid curve** — loss rate increases slowly for short delays, accelerates sharply when shelf life is nearly consumed, then plateaus near 100%. The XGBoost model is trained to reproduce this sigmoid behavior with added realism: noise, interaction effects, and seasonal modulation.

### Training Data
- **Source**: ~50,000 synthetic samples generated by the sigmoid formula with noise
- **Sigmoid label**:
  ```
  consumption_pct = (delay_days / shelf_life_days)^beta × 100 + pre_delay_pct
  loss_rate = L_min + (1 - L_min) / (1 + exp(-k × (consumption_pct - x0)))
  ```
- Beta (1.8) creates a non-linear acceleration effect — shelf life depletion is not linear

### Shelf Life by Temperature (linear interpolation)

| Lettuce Type | 32°F | 41°F | 50°F | 55°F |
|-------------|------|------|------|------|
| Iceberg | 14d | 10d | 7d | 5d |
| Romaine | 14d | 10d | 7d | 5d |
| Butterhead | 14d | 10d | 7d | 3–4d |
| Leaf | 10d | 7d | 5d | 3d |
| Spring Mix | 10d | 7d | 5d | 3d |

### Features (10)

| Feature | Source |
|---------|--------|
| `delay_hours` | From Model 1 (or Pre-Model gate) |
| `delay_temp_f` | `temp_max_mean` from weather features |
| `shelf_life_days` | Interpolated from lettuce type + temperature |
| `consumption_pct` | Calculated from delay, shelf life, and beta |
| `lettuce_type_enc` | Integer encoding (iceberg=0 … spring_mix=4) |
| `organic` | 0 or 1 |
| `month` | From departure date |
| `seasonal_price_index` | Lookup by (lettuce_type, month) — seasonal multiplier |
| `transit_distance_km` | Route distance |
| `pre_delay_consumption_pct` | Shelf life consumed before departure (default 0%) |

### Output
```
loss_rate          # XGBoost predicted fraction (0–1)
sigmoid_loss_rate  # Baseline sigmoid formula (for comparison)
shelf_life_days    # Temperature-adjusted shelf life
consumption_pct    # % of shelf life consumed by the delay
seasonal_price_index
```

### Revenue Loss Calculation

```
price_per_lb           = BASE_PRICES[(lettuce_type, organic)]
total_value            = quantity_lb × price_per_lb
revenue_loss           = total_value × loss_rate
revenue_loss_seasonal  = revenue_loss × seasonal_price_index
```

### Base Prices (per lb)

| Type | Conventional | Organic |
|------|-------------|---------|
| Iceberg | $1.84 | $5.14 |
| Romaine | $3.04 | $6.65 |
| Butterhead | $3.34 | $7.32 |
| Leaf | $2.89 | $6.33 |
| Spring Mix | $3.95 | $8.66 |

- `predict_loss.py` — `LossPredictor` class
- `build_loss_model.py` — training script
- `loss_model_xgb.joblib` — trained artifact (1.3 MB)

## Stage 4 — Supply Risk Assessment

### What
Assesses the broader supply chain risk for the lettuce type and shipment month — independent of the specific route's weather. This reflects whether the *source region* faces seasonal scarcity, historical disaster risk, or production troughs.

### Why
Even if the transit goes smoothly, a shipment may face problems if the source region is in a low-production period or historically disaster-prone month. This layer adds a supply-side perspective to complement the weather-side delay analysis.

### Data Sources
- **USDA production data** — lettuce production volumes by month, used to derive supply index (0–1, where 1 = peak supply)
- **FEMA disaster declarations** — historical frequency of weather/natural disasters per region per month
- Combined in `data/supply_risk_lookup.csv` (5 types × 12 months = 60 entries)

### Harvest Pattern
- **April–October**: Primary source is Salinas Valley, CA (~200 km from Tracy depot)
- **November–March**: Primary source is Yuma, AZ (~800 km from Tracy depot)

This affects both `source_region` in the output and `transit_distance_km` when auto-filled.

### Risk Escalation Logic
Base risk from the lookup is escalated one level if:
- `delay_hours > 48` (extended delay increases regional disruption exposure), OR
- `temp_f > 50°F` (warm transit temperature signals potential cold chain failure)

```
low → medium   (if either condition met)
medium → high  (if either condition met)
```

### Output
```
risk_level         # "low" | "medium" | "high"
supply_index       # 0–1 (seasonal production level)
source_region      # e.g. "Salinas Valley, CA"
disaster_frequency # avg disasters/year in that region/month
details            # human-readable explanation string
```

### Files
- `recommendations.py` — `get_supply_risk()` function
- `data/supply_risk_lookup.csv` — lookup table
- `data/build_supply_index.py` — builds the lookup from USDA + FEMA data

## Stage 5 — Recommendations Engine

### What
Generates 1–4 plain-language, actionable recommendations for the shipment planner, based on the combined outputs of loss rate, delay, temperature, and supply risk.

### Why
The numeric outputs (loss_rate = 0.42, delay_hours = 18.3) are not immediately decision-ready. The recommendations engine translates these into operational actions specific to Costco's supply chain context (reefer truck fleet, backup suppliers, distribution center inventory).

### Logic (independent trigger groups, each evaluated separately)

**Loss-rate triggers:**

| `loss_rate` | Level | Action |
|-------------|-------|--------|
| > 50% | CRITICAL | Postpone shipment + alert backup supplier and adjust receiving schedule |
| 30–50% | WARNING | Reduce shipment size to `quantity × (1 - loss_rate)` lb + notify store |
| 15–30% | WARNING | Monitor weather, keep contingency routing ready |

**Delay triggers:**

| `delay_hours` | Level | Action |
|--------------|-------|--------|
| > 48h | WARNING | Consider alternate route or split shipment across departure windows |

**Supply risk triggers:**

| `risk_level` | Level | Action |
|-------------|-------|--------|
| high | WARNING | Source from alternate growing region, pre-order inventory buffer |
| medium | INFO | Monitor regional crop availability reports |

**Default (nothing triggered):** INFO — "Conditions favorable. Proceed with standard shipment schedule."

### Files
- `recommendations.py` — `generate_recommendations()` function

## Key Design Decisions

**Why a pre-model classifier instead of just using Model 1 everywhere?**
Model 1 is regression over synthetic data. Running it on mild, clear-weather days would produce small but nonzero delay predictions that are noise. The pre-model is trained on real weather event records and explicitly filters out days when no severe weather occurred — making the system more conservative and interpretable.

**Why synthetic training data for Model 1?**
Real historical delay data at the route-segment level doesn't exist for Costco's private fleet at this granularity. Synthetic data generated from known weather-delay relationships (validated against NOAA records) allows training a model that encodes meteorological domain knowledge while still generalizing across route geometries.

**Why XGBoost for Model 2 instead of keeping the sigmoid formula?**
The sigmoid formula is a good first-principles baseline, but it can't capture interactions — e.g., that organic spring mix at 50°F in August has different decay dynamics than conventional iceberg at the same conditions. XGBoost trained on sigmoid-generated labels with noise learns to reproduce the core decay behavior while also fitting interaction terms that pure sigmoid cannot represent.

**Why aggregate pitstop probabilities into route-level KPIs?**
Individual pitstop predictions are noisy (weather station proximity varies). Aggregating across all pitstops on a route smooths out noise and gives a route-level severity signal. The two-condition rule (≥20% of pitstops OR any pitstop ≥70%) is designed to catch both distributed moderate risk and localized extreme risk.

**Why segment_start_hour matters**
A truck that departs at 8 AM will reach different segments at different times of day. A mountain pass segment reached at 11 PM deserves a higher night penalty than if it were reached at noon. Tracking cumulative drive time per segment makes the delay model time-aware in a physically meaningful way.
